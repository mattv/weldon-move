# Working Agreements

## Operating model
- The **GitHub Project board is the source of truth.**
- Every meaningful task is a **GitHub Issue**; large work is an **Epic** issue.
- Every issue has **one owner**.
- README stays short. Docs support the board — they don't replace it.

## How to work here
- **Ship artifacts over discussion.** Create issues, labels, commits, PRs —
  don't produce long plans. ("Stop talking and do.")
- Keep issues **small** (one sitting where possible) and action-oriented:
  Why → Success Looks Like → Checklist → Owner.
- Suggest automation first; keep documentation current; surface risks/blockers.
- Optimize for **reducing stress**, not adding process.

## Issue conventions
Labels are grouped: **Area** (🏠 Parkmeadow, 🚜 Weldon, 📦 Moving, …),
**Owner** (👤 Matt, ❤️ Angel, 🏡 Riley, 🚚 Kavan, …),
**Priority** (🔥 Critical → 🟢 Low), **Status/Special** (❓ Decision Needed, …).

## Board columns
📥 Inbox → 🎯 Ready → 🚧 Working → ⏳ Waiting → 👀 Review → ✅ Finished

## Repo boundary
Anything invented (product/SaaS/strategy) → **private HomeOps repo**.
Anything proven in this real move → stays here.
